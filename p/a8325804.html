【15445】Lab1 Buffer Pool

发表于2022-07-11|更新于2022-07-11|15445
|阅读量:

实验文档

https://15445.courses.cs.cmu.edu/fall2021/project1/

构建一个面向磁盘的存储管理器，buffer pool 负责将物理页面在内存和磁盘之间来回移动。

实验内容

LRU REPLACEMENT POLICY

需要修改的文件有：

src/include/buffer/lru_replacer.h

src/buffer/lru_replacer.cpp

在头文件里面，我们可以看到，已经写好了一些函数声明。

  • Victim(frame_id_t*)：在 lru 链表中删除一个元素，也就是最久未使用元素被剔除了，通过指针返回删除的页面。
  • Pin(frame_id_t)：表示这个页面正在被使用，因此，不能放在替换链表中。
  • Unpin(frame_id_t)：表示这个页面没有被使用了，那么它应该被加入到替换链表中。
  • Size()：返回替换链表的长度。

头文件中，我们需要添加一些数据结构，首先是链表，然后是哈希表（得到页面号和链表位置的映射），然后是锁，保证并发的正确性。最后还有 lru 链表的最大长度。

最后在 lru_replacer.cpp 文件中实现那几个函数。

最后，在 test/buffer/lru_replacer_test.cpp 中去掉 DISABLED_ 然后测试即可。

BUFFER POOL MANAGER INSTANCE

缓冲池管理器负责获取数据库页面并将他们存储到内存中，当需要逐出页面为新页面腾出空间时，还可以将脏页面写入磁盘。

我们需要在 src/buffer/buffer_pool_manager_instance.cpp

中实现定义在 src/include/buffer/buffer_pool_manager_instance.h 中的函数。

一点理解

我们所有的数据都以页为单位，它在磁盘上的位置呢，我们就用 page_id 来作为索引唯一确定。我们的 buffer pool 在内存中，一共分配了 pool_size 个页面，这里我们用 frame_id (0≤frame_id<pool_size0 \le frame\_id<pool\_size0≤frame_id<pool_size) 来作为索引唯一确定。因此，pages_ 是通过 page_id 找到页面在内存中位置 frame_id 的哈希表。

FlushPgImp

给定页面，将它刷新到磁盘中。首先判断为有效页面并且在内存中了，然后找到在内存中的位置，然后写回，并且设置为非脏。（或者判断为脏才写回，否则无动作）。

FlushAllPgsImp

同上，不过只是处理内存中的全部页面。

NewPgImp

在内存中腾出一个全新的页面，然后绑定一下磁盘中的某个位置 （new_page_id）。

FetchPgImp

获取（使用）一个页面，如果内存中有，那么直接返回。如果没有，就从磁盘中写入内存。注意 pin 的数量要增加。

DeletePgImp

从内存中删除页面，如果 pin 不为 0 的话那么无法删除。

UnpinPgImp

取消使用，pin 减少，如果减少到 0 了，需要在 replacer 中添加它。使用完成后如果脏页需要标记。

PARALLEL BUFFER POOL MANAGER

注意在头文件中添加需要的头文件。其实就是比如说 100 个页面，有 5 个 BufferPoolManagerInstance ，那么它们就依次负责 1、6、11 …… 等。这样既可以实现并发，又不会因为锁而浪费太多时间。
